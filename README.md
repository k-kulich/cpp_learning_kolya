## Работа с Git

Мы находимся на гитхабе, но это совсем не то же самое, что Git. __Git__ - это система контроля версий, то есть специальная утилита, которая позволяет запоминать изменения в файлах и откатываться к какому-то из "сохраненных" вариантов. Почему "сохраненных"? Потому что по факту git не хранит копии и старые версии файлов, весь этот эффект основан на хэшах (специальным образом чиселки считаются и обозначают разницу между версиями), потому что иначе такая "система контроля" была бы крайне дорогой по памяти, а все ресурсы для нас крайне важны. Так что все устроено немного сложнее. А вот __Github__ - это просто конкретная платформа, предоставляющая удобный интерфейс работы с git, а заодно своеобразную соцсеть. 

Итак, для начала, тебе нужно скачать git на свой комп. Вообще трушно с ним, конечно, работать через терминал, точно также, как и компилировать программы. Скачивай с сайта, чтоб тебе не искать долго: ``.

Потом на компе в удобной тебе папке делаешь:

```bash
git clone ...
```

Все, теперь там наш проект заспавнится, и не надо каждый раз на github лезть. Ясное дело, через VS Code можно вообще на изи работать с git, но я покажу, как сделать это через терминал, потому что навык работать без GUI реально очень ценится.

Надо бы еще для удобства добавить краткое имя для нашей удаленной репы, чтобы каждый раз полный url не вводить:
```bash
git origin ...
```

Теперь, чтобы подгрузить изменения, которые я туда залила, тебе достаточно выполнить такую команду:

```bash
git pull
```

После этого все изменения будут подгружены. Не забывай делать его перед занятиями, чтобы у нас с тобой версии не супер сильно отличались.

Дальше - больше. Теперь было бы хорошо тебе самому свой код тоже сюда загружать, поэтому разберемся с тем, что тут да как. Чем, собственно, еще приколен git - а тем, что нам не надо париться о том, что мы вот сейчас фичу делаем, а она сразу недоделанная в прод попадает, потому что мы можем создавать разные ветки - буквально ответвление от основной истории изменений. 

Зачем это нам? а затем, чтобы мы потом могли большие проекты писать, а не только те, где пара файликов нужна. Ну и чтобы ты спокойно делал ДЗ в своем темпе, и мог залить, когда уже доделаешь, и если вдруг (вдруг!!!) это случилось позже, чем ожидалось, то ты не переживал на тему еще не рабочего кода. 

Итак, ветки. Создаются новые ветки с помощью команды:
```bash
git branch <название новой ветки>
```
Тогда ветка просто создается от текущей версии проекта, и нам нужно в нее перейти:
```bash
git checout <название ветки, куда хотим перейти>
```

Супер, можем в ней работать теперь! По умолчанию в каждом репозитории создается ветка `main` и в ней ведется работа. Так вот, у нас в `main` буду работать __только я__. Это будет для того, чтобы тебе было нужно именно отправлять ДЗ - они не просто спавнились у меня в репе, а создавался pull request, который мне надо было бы одобрить. Так будет лучше и правильнее: в проде сразу никто не пишет, все всегда все пихают по веткам, а уже потом объединяют как раз пул реквестами. Так что создавай свою ветку, называй как-нибудь и работай в ней :)

Собственно, как сохранять версии то?

1. Добавляем файлы и папки, которые надо отслеживать:  `git add относительный/путь/до/файла/или/папки`
2. Запоминаем изменения (коммитим): `git commit -m "подпись для коммита"`. Так-то можно и без подписи, но лучше с ними: так понятнее и тебе и другим, каие именно изменения и куда были внесены. 
3. Надо отправить все на сервер, чтобы изменения дошли до меня и я смогла их к себе выгрузить: `git push` или `git push origin`. Теперь та ветка, в которой ты работал, запушится, и я все увижу.

А вот пул реквест он сам тебе предложить сделать через github, как только ты загрузишь изменения в облако. Там достаточно протыкать кнопочки и добавить какое-никакое описание.

! Важно: делать merge нельзя! как его делать, расскажу потом, но тебе нужно дотыкать по зеленым кнопкам только до момента `Create pull`, после этого никакие кнопки не нажимаем и возвращаемся, иначе весь смысл пула будет потерян - ты сам и примешь все изменения, пропуская этап проверки ДЗ.

## Установка компилятора (g++)

### Windows 11

Рекомендуемый способ – установить **MSYS2** с пакетом `mingw-w64-ucrt-x86_64-gcc`.  
Это даст тебе современный компилятор GCC и утилиты командной строки (bash, make и т.д.).

1. Скачай установщик MSYS2 с официального сайта: [https://www.msys2.org/](https://www.msys2.org/)
2. Запусти установщик и следуй инструкциям. По умолчанию MSYS2 устанавливается в `C:\msys64`.
3. После установки запусти **MSYS2 UCRT64** из меню «Пуск» (или «MSYS2 MSYS» – UCRT64 предпочтительнее).
4. В открывшемся терминале выполни обновление пакетов:
   ```bash
   pacman -Syu
   ```
   При необходимости перезапусти терминал и повтори команду.
5. Установи компилятор:
   ```bash
   pacman -S mingw-w64-ucrt-x86_64-gcc make
   ```
6. Проверь, что установка прошла успешно:
   ```bash
   g++ --version
   ```
   Ты должен увидеть версию GCC (например, 13.2.0).

Теперь ты можешь использовать **g++** из этого терминала. Для работы с файлами перейди в нужную папку командой `cd /путь/к/проекту` или открой терминал прямо в папке через контекстное меню (если у вас настроено).

> **Примечание:** Не используй обычную командную строку Windows (CMD) или PowerShell для запуска g++, если ты не добавил путь вручную. Проще запускать именно терминал MSYS2.

### Arch Linux

В Arch Linux компилятор g++ устанавливается через пакетный менеджер `pacman`.

1. Открой терминал.
2. Выполни команду:
   ```bash
   sudo pacman -S gcc make
   ```
3. Проверь версию:
   ```bash
   g++ --version
   ```

## Компиляция программы

Предположим, у тебя есть один или несколько `.cpp` файлов.  
Базовая команда для компиляции:

```bash
g++ -std=c++17 -o имя_выходного_файла исходник1.cpp исходник2.cpp ...
```

- `-std=c++17` – стандарт языка (можно использовать `-std=c++20`, если хоти).
- `-o имя_выходного_файла` – задаёт имя исполняемого файла.
- После `-o` перечисляются все файлы с исходным кодом.

**Пример для одного файла `main.cpp`:**
```bash
g++ -std=c++17 -o program main.cpp
```

**Пример для нескольких файлов:**
```bash
g++ -std=c++17 -o AnalyzeLog main.cpp parser.cpp stats.cpp
```

После успешной компиляции появится исполняемый файл:
- В Windows – `program.exe` (или `AnalyzeLog.exe`).
- В Linux – `program` (или `AnalyzeLog`).
